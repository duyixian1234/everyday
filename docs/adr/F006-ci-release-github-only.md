# ADR F006: CI + GitHub-only release workflow (cnb mirror excluded)

**Status:** Accepted
**Date:** 2026-07-10

## Context

Everyday targets three desktop platforms (Linux, macOS, Windows) plus Apple Silicon macOS as an explicit extra slice. Releases produce prebuilt binaries attached to a GitHub Release. A second remote exists (`cnb.cool`) as a mirror but has no usable credentials in the project's automation context.

A few decisions had to be explicit before the release flow could be trusted:

1. Which CI platform runs quality gates?
2. Which release platform builds and publishes binaries?
3. Is the mirror auto-pushed or manual-only?

## Decision

### CI: GitHub Actions only

`.github/workflows/ci.yml` runs on every push and PR:

- Three matrix targets: `ubuntu-latest`, `macos-latest`, `windows-latest`.
- Steps: `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, `cargo test`, `cargo build`.
- Clippy `-D warnings` is non-negotiable: any warning fails the build.

A second matrix job adds `macos-latest` with target `aarch64-apple-darwin` to cover Apple Silicon compilation in CI even when the runner can't execute the binary natively.

### Release: tag-driven GitHub Actions

`.github/workflows/release.yml` triggers on `v*` tag push:

- Builds four targets: `x86_64-unknown-linux-gnu`, `x86_64-apple-darwin`, `aarch64-apple-darwin`, `x86_64-pc-windows-msvc`.
- Uploads the artifacts to a GitHub Release tied to the tag.
- The release notes are auto-generated by `softprops/action-gh-release`.

### Mirrors: GitHub only for pushes and tags

- The `origin` remote is `git@github.com:duyixian1234/everyday.git` (SSH) in this clone.
- The `cnb` remote (`https://cnb.cool/c10ve/everyday`) is **not pushed** during releases. There are no usable credentials, and pushing tags to it would orphan the release.
- The release runbook (`progress.md`, the long-term memory file) spells out: `git push origin master && git push origin vX.Y.Z`.

### Release commit convention

- One feature commit (no version bump).
- One `chore: release vX.Y.Z` commit that bumps `Cargo.toml` and updates docs.
- Tag is annotated: `git tag -a vX.Y.Z -m "vX.Y.Z: <highlights>"`.

## Alternatives considered

### Cross-platform CI via a third-party service (Cirrus, Buildkite)

- Pro: cheaper runners on some platforms.
- Con: GitHub Actions is already integrated with the repo's auth model, secrets, and PR UI.
- Rejected: zero value from the swap.

### Push the cnb mirror as part of the release

- Pro: keeps the mirror up to date.
- Con: the mirror has no usable automation credentials in this environment.
- Rejected: mirrors are updated manually when needed.

### Auto-tag the release from CI

- Pro: one-step release.
- Con: requires granting CI write access to tags; complicates cherry-pick releases and backports.
- Rejected: tagging stays a human step.

### Single combined CI+Release workflow

- Pro: less YAML to read.
- Con: re-runs the full matrix on every tag, which is wasteful.
- Rejected: separate workflows match the separate triggers.

## Consequences

- Every PR is gated by the same quality bar across three OSes plus Apple Silicon compilation.
- A `chore: release vX.Y.Z` commit + annotated tag is the canonical release recipe; documented in the runbook.
- Mirror clones on cnb.cool are stale until manually synced (and that's fine — cnb is not a release surface).
- Quality gate discipline (`fmt --check`, `clippy -D warnings`, `test`, `build`) is shared by the project, the CI, and the release workflow.
